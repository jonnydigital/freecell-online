### Hub homepage hydration duplication (DOM, not SSR)

**Observed:** Live DOM at `solitairestack.com/` after JS hydration has 12 `application/ld+json` scripts; each of the 6 ids appears exactly twice (2× `ld-organization` in `<head>`, 2× of each hub schema in `<body>`).

**Root cause hypothesis:** the hub `page.tsx` and `(main)/layout.tsx` render `<script>` tags inside the React component tree. Next.js streams the SSR HTML (1× each) AND inserts the same scripts again from the React Server Components payload during client hydration. The April 15 `id=` fix was supposed to let React reconcile, but for `<script>` tags inside the body React 18 does not consistently match by id — the duplicates persist.

**Verification:**

```text
curl -s https://solitairestack.com/ | grep -oE 'id="ld-[a-z-]+"' | sort | uniq -c
# 1 of each id (6 total) — SSR HTML is clean

DOM probe at runtime:
totalScripts: 12, uniqueIds: 6, ids appear in pairs
```

**SEO impact:** none. Google reads SSR HTML, which is correct (1× each). The duplicates only exist in the rendered DOM after JS executes.

**Other impact:** small. Some third-party schema scrapers (e.g. Tag Manager schema validators) walk the DOM and may report doubled scripts as a warning. Wastes ~3KB of post-load DOM for users.

**Not shipped today.** The fix vector is different from April 15's `id=` patch — likely needs to either (a) move JSON-LD into Next.js's `metadata.other` field which uses Float APIs that DO reconcile, or (b) use `next/script` with `strategy="beforeInteractive"`. Out of scope for the small commit window today; carrying forward.
