# Daily QA Report — 2026-05-21

## Summary

Daily review across `playfreecellonline.com`, `solitairestack.com`, and the
`playspidersolitaireonline.com` / `playklondikeonline.com` spokes.

**Headline:** a network-wide canonical regression was caught and fixed. Every
page that inherits the root layout's default canonical — game pages, `/strategy`,
`/how-to-play`, all blog posts, and the homepages — was emitting
`<link rel="canonical">` pointing at its own **domain root** instead of itself,
on **all four domains**. Left in place, Google would have collapsed each entire
site to its homepage and dropped every deep page from the index. The root cause
was commit `e408d21` (2026-05-20), which changed the root layout canonical from
the relative `'./'` to the absolute `absoluteUrl('/')`. Fixed in commit
`328c075` (revert to `'./'`) and verified live on all four domains. A second
commit `bcf90f4` hardens `scripts/audit-canonicals.mjs` so this regression class
fails CI in future.

Visual QA PASS on both primary sites at desktop. Mobile interactive resize
remains blocked the same way as prior cycles; mobile rendering re-verified via
CSS `clamp()` math, the `<=767px` media-query audit, and a live CSS-variable
simulation. Competitor `online-solitaire.com` unchanged at the surface from
May 10.

Two items flagged for human attention: (1) a prompt-injection string embedded in
the competitor's page, ignored; (2) an **uncommitted** working-tree change in

## Phase 3 — SEO & Schema Audit (the canonical regression)

Spot-checking game/content pages surfaced a wrong canonical, which expanded into
a network-wide finding:

- `solitairestack.com/strategy`, `/golf`, `/how-to-play` — all emitted
  `<link rel="canonical" href="https://solitairestack.com">` (homepage),
  confirmed in the **raw SSR HTML** (what Google crawls), not just runtime.
- `playfreecellonline.com/` `/game/1` `/bakers-game` `/eight-off`
  `/freecell-for-beginners` — all → `https://playfreecellonline.com` (root).
- `playspidersolitaireonline.com/spider` → its root;
  `playklondikeonline.com/klondike` → its root.

Every page on every domain canonicalised to its own domain root. The May 10
report recorded `/strategy` as correctly self-canonical, so this was a
regression introduced in the May 10 → May 21 window.

**Root cause:** commit `e408d21` (2026-05-20) changed `src/app/layout.tsx`
`metadata.alternates.canonical` from `'./'` to `absoluteUrl('/')`. The relative
`'./'` is resolved by Next.js per-page against `metadataBase` + pathname, giving
each page a self-referential canonical. The absolute `absoluteUrl('/')` hard-codes
the homepage URL for every page that inherits the layout default. Pages that set
their **own** `alternates.canonical` (e.g. `/freecell-vs-spider`,
`/microsoft-freecell`, `/freecell`, `/freecell-solvability`) were unaffected —
confirmed still correct in SSR.

Other schema spot-checks: H1 counts are 1 per page on all pages checked;
JSON-LD types present (Organization, WebApplication, Article/Game/HowTo,
FAQPage, BreadcrumbList). The known hub JSON-LD hydration duplication
(SSR-clean, runtime-doubled, e.g. `/golf` 6→12) persists and remains a cosmetic
non-SEO backlog item. **Note:** `aggregateRating` was deliberately *not* re-added
during this audit — commit `e408d21` removed fabricated ratings for E-E-A-T
integrity, and that decision is respected here (see Phase 4 flag).

## Addendum — second run (15:58), disposition of the flagged `page.tsx` change

A second scheduled-task run overlapped this session and picked up the open
"pending human decision" flag on the uncommitted `page.tsx` `aggregateRating`
change. Disposition: **reverted, not shipped.**

Verification performed: traced the rating values to their source. The
`StarRatingWidget` (`src/components/StarRatingWidget.tsx`) derives its displayed
rating/count entirely from **hardcoded props** — `baseCount` / `baseAvg`
defaults (3241 / 4.8) and the per-site overrides in `FreecellBelowFold`
(default), `KlondikeBelowFold` (1876 / 4.7) and `SpiderBelowFold` (2184 / 4.7).
There is no real aggregation: every visitor sees the same seeded numbers, plus an
optional +1 from their own `localStorage` rating. These are **synthetic counts**.

Per this report's own Phase-4 guidance ("if the counts are still synthetic,
re-adding them re-opens the exact integrity flag `e408d21` closed"), the change
was reverted with `git checkout -- src/app/(main)/page.tsx`. Working tree is
clean; nothing committed for the `aggregateRating` line of work. The canonical
regression fix and audit guard from this cycle remain shipped and pushed.

Recommendation: keep `aggregateRating` JSON-LD **off** all pages until a genuine
ratings source exists (real per-game user votes persisted server-side). At that
point the change is a ~10-line edit and becomes legitimate. Until then, emitting
star markup over invented counts is a Google spam-policy / E-E-A-T risk and a
trust issue with searchers — not worth the CTR. This closes the flagged item; no
further human decision required unless real ratings are introduced.
