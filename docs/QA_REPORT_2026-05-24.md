chain, not in card-width `clamp()` sizing.

### Mobile + Tablet — CRITICAL BUG FOUND → FIXED → VERIFIED

**Symptom.** At 375 / 390 / 414 / 768 px, `playfreecellonline.com` and
`solitairestack.com` rendered the top bar and the Menu/New/Undo/Redo/Hint toolbar,
then a fully blank green board below. DOM probing showed all 52 `.dom-card`
elements present (opacity 1, visibility visible, white background, valid
`getBoundingClientRect` positions) yet **not painted** — `elementsFromPoint` at
each card's center returned only the page-root green divs, never the card.

**Root cause.** Pure flexbox height-collapse on the FreeCell shell
(`src/components/dom-freecell/DomGameShell.tsx`):

- The outer game shell (`[data-scroll-role=game-sticky-shell]`) used
  `flex-1 h-[100dvh]`. On desktop the parent row (`lg:flex-row`) makes the shell's
  height definite, so `h-full` children resolve correctly.
- On mobile the parent becomes a flex **column** (`flex-col`), so the shell's
  height is produced by `flex-grow` (flex-basis `0%`), which is **indefinite** for
  percentage-height resolution.
- Consequently the inner shell (`flex h-full flex-col overflow-clip`) collapsed
  to ~105px instead of filling the viewport, the `flex-1` board region collapsed
  to **0px**, and the `absolute inset-0 overflow-clip` board surface (+ all cards)
  was clipped away — a blank board.

Measured before fix at 375px: inner shell height 105px, board region height
**0px**.

**Fix (commit `6fc3f3b`).** Scope the grow to the desktop row only:
`flex-1` → `lg:flex-1`. On mobile the shell now keeps its explicit
`height: calc(100dvh - 1rem)` (definite), so `h-full` resolves and the board fills.
Desktop is provably unchanged (`lg:flex-1` === `flex-1` at ≥1024px). The fix was
validated on the live DOM *before* editing code (applying `flex: 0 1 auto` to the
shell took the board region from 0 → 691px and rendered the full board), then
applied, `tsc --noEmit` clean, committed, pushed, and re-verified post-deploy.

**Post-fix verification (live production, headless true-viewport):**

| Viewport | Board region height | Cards in viewport |
|---|---|---|
| playfreecell 375 | 691px | 52 |
| playfreecell 390 | 723px | 52 |
| playfreecell 414 | 775px | 52 |
| playfreecell 768 (tablet) | 890px | 52 |
| solitairestack 375 | 691px | 52 |
| eight-off 375 (FreeCell variant) | 691px | 52 |

Visual confirmation: 4 free cells + 4 foundations + 8 readable card columns + top
bar + bottom toolbar all render correctly on mobile.

### Scope check — other engines unaffected

Klondike (`playklondikeonline.com`), Spider (`playspidersolitaireonline.com`), and
the generic-engine games (e.g. Pyramid) use different shells
(`DomKlondikeShell`, `DomSpiderShell`, `GenericSolitaireShell`) that do **not** use
the `flex-1 h-[100dvh]` + `h-full` percentage pattern. Mobile-tested Klondike and
Pyramid at 375px: both render their boards normally (non-zero board surface, cards
in viewport). The bug and the fix are isolated to the FreeCell-family shell.
