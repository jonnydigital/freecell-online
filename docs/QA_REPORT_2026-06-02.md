### JSON-LD audit — confirmed: known issue still present

| Page (live) | SSR HTML scripts | Runtime DOM scripts | Status |
|-------------|------------------|---------------------|--------|
| `solitairestack.com/freecell` | 5 | 5 | clean |
| `solitairestack.com/tripeaks` | 6 | **12** | duplicated 2× |
| `solitairestack.com/pyramid`  | 5 | **10** | duplicated 2× |
| `solitairestack.com/golf`     | 6 | **12** | duplicated 2× |
| `solitairestack.com/klondike` | 7 | **12** | duplicated 2× |
| `solitairestack.com/spider`   | 6 | **12** | duplicated 2× |

**Mechanism (re-confirmed today):** SSR HTML for each affected page contains exactly one of each schema (verified by raw `curl` against the live URLs — five unique `id="ld-…"` script tags in the served HTML). After hydration, the runtime DOM shows each schema duplicated. Each unique `id` value appears exactly twice in the rendered DOM, both copies byte-for-byte identical, both in `<body>` (except `ld-organization` which is duplicated in `<head>`). The `JsonLd` component (`src/components/content/JsonLd.tsx`) assigns a stable `id` exactly to prevent this, but the duplication still happens on every page that uses the component as a sibling of a `'use client'` subtree (`GenericGamePage` is the common factor on /tripeaks, /pyramid, /golf, /klondike, /spider; `/freecell` is unaffected because it inlines raw `<script>` tags directly in JSX and uses a different client wrapper).

**Impact:** moderate — Google tolerates byte-identical duplicate JSON-LD, the second copy is wasted bytes, and Search Console URL Inspection may flag it. Not currently believed to risk a manual action.

**Fix path (still architectural backlog):** migrate page-level schemas off the `JsonLd` component to either (a) `metadata.other` so Next.js renders them once in `<head>`, or (b) `next/script` with `strategy="beforeInteractive"`. Either path touches every game page that uses the component, so it stays out of a single-cycle commit.
