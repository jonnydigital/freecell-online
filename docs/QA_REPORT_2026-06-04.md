### Mobile (375 / 390 / 414px) — BLOCKED for live screenshots; PASS at code level
The browser window could not be narrowed this run:
- `resize_window` (Chrome extension) returned success but `window.innerWidth` stayed **1500px**.
- AppleScript `set bounds` and `set zoomed to false` on the game window were both ignored — bounds stayed `0,121,1512,982`. A window manager is clamping Chrome to full screen width.

This is the same environmental clamp noted in prior reports (May 24 onward). Substituted a source review of `src/components/dom-engine/dom-card-styles.css`:
- `@media (max-width: 767px)` sets `--card-width: clamp(34px, (100vw - 44px) / 8, 54px)`.
- At 375px viewport → `(375-44)/8 = 41.4px` → **~41px**, squarely in the healthy 30–54px band the QA spec expects.
- `overflow: hidden` appears **only** on the confetti container (full-viewport) and `overflow: visible` on the sparkle layer — neither clips cards.
- The May 24 mobile blank-board fix (flex height scoping to `lg`) is structurally intact in source.

Conclusion: no detectable mobile regression, but a true-viewport visual confirmation is owed once the window can be resized (or via a headless true-viewport pass). Filed as carry-forward.
