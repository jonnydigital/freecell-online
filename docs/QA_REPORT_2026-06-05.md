### Mobile / Tablet — true-viewport screenshots BLOCKED (tooling)
The Chrome automation render viewport is **pinned at 1500×737** and cannot be lowered in this environment:
- `resize_window` reports success and the **OS window did physically shrink** (System Events confirmed 800×812, then 606×812), but the page's `innerWidth` / `visualViewport.width` stayed **1500**.
- The viewport meta is correct (`width=device-width, initial-scale=1, viewport-fit=cover`), so this is not a site bug — it is a CDP device-metrics override on the automation browser that the resize path does not clear.

**Mobile verified by source review instead.** `src/components/dom-freecell/dom-card-styles.css` `@media (max-width: 767px)` sets:
```
--card-width: clamp(30px, calc((100vw - 61px) / 8), 54px);
```
At 375 / 390 / 414px this yields **~39 / 41 / 44px** cards — inside the healthy 30–54px band — and the viewport-relative formula mathematically guarantees all 8 columns + gaps + padding fit within `100vw` (no horizontal overflow by construction). `overflow:hidden` appears only on confetti/sparkle layers, never on card containers. The May 24 mobile blank-board regression remains structurally fixed.

> **Recommendation:** add a headless true-viewport screenshot step (Playwright/Puppeteer with an explicit `viewport` + `deviceScaleFactor`) to the QA harness so phone/tablet rendering can be captured visually rather than inferred from CSS. The interactive Chrome extension cannot reach sub-1500px here.
