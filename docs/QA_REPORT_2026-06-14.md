### Window-resize trap — re-confirmed (consistent with Jun 11/12 notes)
`resize_window` shrinks the **outer** Chrome window (outerWidth dropped to 784) but the **render viewport stays pinned at 1512px** (`innerWidth == visualViewport == screen.width`), so `@media` queries never fire and a screenshot shows the full desktop layout at any requested size. This produces convincing-but-false "no mobile layout" / "overflow" readings. The reliable method is the **same-origin iframe rig**: navigate the parent tab to the target origin, inject an iframe at a true 375/390/768px width, and measure/inspect inside it. Cross-origin iframes are X-Frame-Options–blocked, so the parent tab must already be on the same origin as the iframe src.

## 4. Bug found, fixed, shipped — mobile overflow on wide cascade boards

**Symptom:** On phones, games with **more than 8 columns** — Forty Thieves (10), Bristol (11) — overflowed horizontally. Measured on the true-viewport iframe rig at 384px: Forty Thieves board content was **490px wide inside a 352px board surface** (`boardScrollWidth 490 > clientWidth 352`), with cards stuck at a 60px floor.

**Root cause:** `src/components/dom-generic/GenericCascadeBoard.tsx` set `--card-width` **inline** for >8-column layouts as `clamp(60px, calc((100vw - 288px) / divisor), 100px)` — a desktop-tuned formula with a 60px minimum. Because an inline style beats the `@media (max-width: 767px)` rule in `dom-card-styles.css`, mobile was pinned to the 60px floor → 10 columns × 60px ≈ 600px content, far past a 375–414px viewport. The nice mobile clamp the 8-column boards use (`clamp(34px, (100vw-44px)/8, 54px)`) was being bypassed for exactly the games that most needed it.

**Fix (commit `2fe2d39`):** Stop setting `--card-width` inline. The board now exposes `--cascade-count` and `data-wide="true"`, and new responsive CSS rules `.dom-board-surface[data-wide="true"]` scale card width from the column count at each breakpoint (desktop / tablet 768–1023 / mobile ≤767) — mirroring the existing `[data-game="spider"]` 10-column pattern. The mobile rule is `clamp(22px, calc((100vw - 56px) / var(--cascade-count)), 50px)`.

**Verification (end-to-end):**
- `npx tsc --noEmit` clean.
- `calc()` division-by-variable validated in-browser before push (resolves to a valid px value).
- Deploy `dpl_5HjFcJu53JK8xSsoDBtWYjqeeqqF` reached **READY** (~85s), aliased to `playfreecellonline.com`.
- Re-tested live via the iframe rig @384px: **Forty Thieves** cardW 32.8px, `boardScrollWidth == clientWidth == 352` (overflow eliminated), all 10 columns fit; **Bristol** cardW 29.8px, all 11 columns fit; `data-wide="true"` and `--cascade-count` 10/11 confirmed on production.
- **8-column games (FreeCell, etc.) unaffected** — `data-wide` is absent and the base mobile rule is unchanged; FreeCell mobile re-verified at 375/390 (cardW 38.5px, no overflow).

This is a real, reproducible bug confirmed on a true viewport (not the headless/window-resize phantom that prior runs warned about). The mechanism was traced to a specific line and the fix re-tested on the live viewport that exhibited it.
