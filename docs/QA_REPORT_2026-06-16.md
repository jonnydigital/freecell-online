### Window-resize trap — re-confirmed (consistent with Jun 11/12/14/15)
`resize_window` to 375×812 shrank the OS window (`outerWidth` → 756) but the **render/layout viewport stayed pinned at `innerWidth` 1440** — `matchMedia(max-width:640px)` stayed `false`, so media queries never fire and the screenshot keeps rendering the desktop layout. This is the same false-overflow trap documented on prior runs. The reliable method remains the **same-origin iframe rig**: inject an iframe sized to the target width pointing at the game route on the *same* origin (cross-origin iframes are XFO-blocked, so the parent tab must already be on that origin), then read `contentDocument` for true card metrics. All mobile/tablet results above came from that rig and PASS cleanly.

### Observation (not a layout bug — no code changed)
`solitairestack.com/pyramid` renders a **7-column cascade-style board** (stock + 4 foundations + 7 stacked columns) rather than the canonical 28-card triangular Pyramid layout (pair-to-13 removal). This is a possible game-logic/content mismatch, not a layout/rendering defect — the board displays cleanly with no clipping at every viewport. Flagged for owner verification of the Pyramid engine/ruleset; deliberately **not** touched autonomously since it would be a gameplay-logic change outside visual-QA scope.
