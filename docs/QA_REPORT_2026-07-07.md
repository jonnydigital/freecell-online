## Phase 4 — Fix: French locale `<html lang>` correction (U5 slice)
While running the U5 French-locale completeness audit, found a real defect the prior locale audits (which checked `alternates.languages`/hreflang tags + H1) did not catch: the root layout (`src/app/layout.tsx:140`) hardcodes `<html lang="en">`, so **every localized route inherits `lang="en"`** even though its content is French. Confirmed live on `/freecell-en-francais` (rendered `document.documentElement.lang === "en"`). This is an accessibility defect (screen readers announce French content with an English voice) and a rendered-DOM language-signal defect. Its hreflang set is otherwise complete and reciprocal (fr/en/es/de/it/pt/x-default) and H1/canonical are correct.

**Fix (one small slice — French only per plan):** added `src/components/HtmlLang.tsx`, a tiny null-rendering client component that sets `document.documentElement.lang` to the page's language on mount and restores it on unmount (correct for SPA navigation). Wired `<HtmlLang lang="fr" />` into the FR landing page (`/freecell-en-francais`) and FR play route (`/freecell-en-francais/jouer`).

- **Scope note:** the same defect affects the other five locales (es/de/it/pt landing+play). Left for follow-up runs to keep this a single small slice per the "never a whole unit" rule. A fully-SSR fix (raw-HTML `lang` for crawlers that don't execute JS) would require a middleware-driven root-layout `lang` — noted as backlog below; today's client-side fix corrects the accessibility + rendered-DOM signal immediately at low risk.
