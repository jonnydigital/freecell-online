### Mobile ‚Äî PASS (with one UX defect, fixed)

**Rig limitation and how it was worked around.** `resize_window` moves the OS window but the layout
viewport stays pinned at `innerWidth: 1440`, so media queries never fire ‚Äî this is the same
limitation the 07-10 run hit and worked around by "verifying via source". Source-reading can't catch
layout bugs, so this run tested the **real mobile CSS** instead: the page was loaded into a
**same-origin iframe** sized to each device width, which resolves `@media` queries and
`window.innerWidth` against the iframe box. Measurements below are from live rendered DOM, not source.

| Viewport | Cards | Card W | Clipped | Overflow-X | Unused vertical |
|---|---|---|---|---|---|
| 375x812 | 52/52 visible | 39.25px | 0 | none | **69%** |
| 390x844 | 52/52 visible | 41.13px | 0 | none | **69%** |
| 414x896 | 52/52 visible | 44.13px | 0 | none | **69%** |
| 768x1024 (tablet) | 52/52 visible | ‚Äî | 0 | none | ‚Äî |

All acceptance criteria pass: 52 cards in DOM and visible, widths inside the 30‚Äì54px band, nothing
clipped, top bar (deal #, time, moves) and bottom toolbar (home/shuffle/undo/redo/hint) both visible.

**Defect found ‚Äî mobile portrait wastes 69% of the screen.** Card *width* is correctly pinned by
viewport width (8 columns must fit 375px, so ~39px is forced and not a bug). But card *height* is
`width x 1.4 = 55px`, and `--cascade-overlap` is `clamp(10px, w*0.32, 18px)` ‚Üí only **~12.6px** of
each buried card is visible at 375px. The rank glyph alone is 8px, so the visible strip barely
carries rank + suit ‚Äî while ~562px of screen sits empty below the board. Tight strips on the exact
cards a player must read to plan a move.

**Klondike spoke at 375px ‚Äî not verified.** `playklondikeonline.com` is cross-origin and sends
frame-ancestors, so the iframe rig can't reach it, and the window rig can't shrink. Verified from
shared source instead (Klondike inherits the base mobile overlap). **This is why today's fix is
scoped to FreeCell only** ‚Äî see below. Carrying "mobile rig for cross-origin spokes" as backlog.

## Phase 4 ‚Äî Improvements shipped

**1. FreeCell mobile portrait: deepen cascade overlap (U6 play-UX slice).**

- `src/components/dom-freecell/DomBoard.tsx` ‚Äî add `data-game="freecell"` to the board surface,
  matching the existing convention Spider already uses (`data-game="spider"`). No behavioural change
  on its own; it's the scoping hook.
- `src/components/dom-freecell/dom-card-styles.css` ‚Äî new
  `@media (max-width: 767px) and (orientation: portrait)` rule scoping
  `--cascade-overlap` to `clamp(13px, calc(var(--card-width) * 0.42), 24px)` for
  `[data-game="freecell"]`. At 375px each buried card now shows **~16.5px instead of ~12.6px** (+31%),
  spending free vertical space on the strip that carries the rank and suit.

Why it's scoped this narrowly ‚Äî `--cascade-overlap` is a **shared** variable consumed by
`DomFreecell`, `DomKlondikeBoard`, `DomSpiderBoard` and `GenericCascadeBoard` across all four
domains. Editing the base mobile rule would have silently changed Klondike, Pyramid, TriPeaks and the
generic boards ‚Äî none of which I could verify at mobile width today (cross-origin, see Phase 1). So
the change is scoped to the FreeCell family (the one board I can actually measure) and to portrait
only, since landscape mobile is already vertically tight and is special-cased in `DomGameShell`.

Headroom check: deepest realistic column (~19 cards) = 55 + 18x16.5 ‚âà 352px from a tableau top of
120px ‚Üí 472px, comfortably inside 812px. No vertical overflow introduced.

Validation: `npx tsc --noEmit` clean; `npm run lint` **0 errors** (447 pre-existing warnings, allowed).
