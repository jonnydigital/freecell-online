### Mobile — 🔴 P0 FOUND

| Viewport | FreeCell | Spider | Klondike |
|---|---|---|---|
| 375×812 | ✅ 52/52 cards, 39.3px wide, no clip, no h-scroll | 🔴 **4-Suit chip untappable** | 🔴 **D3 chip untappable** |

| 768×1024 (tablet) | ✅ 52/52, 74.4px, board fills without sidebar | — | — |

FreeCell's top status strip and bottom control bar (Menu / New / Undo / Redo / Hint) render
correctly at every width tested.

**The bug.** The mobile top strip added in `e93b4b8` is a single flex row holding three things:
the game switcher (133px), the mode chips (98px) and the stats pill (~157px). At 375px the row has
355px of usable width for ~406px of content. The pill is `flexShrink: 0` and the left group has
`minWidth: 0`, so the left group shrank below its content and its chips overflowed **underneath**
the pill rather than being clipped or wrapped.

Measured on Spider at 375px:

- 4-Suit chip: x 219 → 249. Stats pill: x 233 → 275. Overlap 16px × 11px.
- `document.elementFromPoint()` at the chip's centre returned the **timer `<div>`, not the button** —
  i.e. the tap does not reach the control. Same failure on Klondike's D3 chip (x 190 → 224).
- Not detectable as overflow: `scrollWidth === innerWidth`, no horizontal scroll, no offscreen
  buttons. Only a hit-test finds it. (Assertion now worth adding to the audit harness — see backlog.)

This bites the single most common phone class (iPhone SE / 8 / X / XS / 11 Pro / 12–13 mini all
report a 375px CSS viewport), and it blocks the mode people specifically come to a Spider site for.

## Phase 2 — Competitor (online-solitaire.com)

Unchanged for the 31st+ consecutive run on desktop. One genuinely useful new observation, captured
at 375px: **their entire mobile top bar collapses to a single 32×40 hamburger.** The Turn 1 / Turn 3
chips and the stats readout are removed from the mobile bar entirely and live behind the menu — zero
occluded controls.

That is independent confirmation that switcher + chips + stats cannot coexist on one row at 375px.
They solved it by burying the mode selector; we solved it by giving the chips their own row, so we
keep one-tap mode switching where they cost a menu open. Logged to `docs/research/competitor-log.csv`.

## Phase 4 — Improvements shipped

1. **`c2bb372` — fix(ux): stop the mobile stats pill from swallowing the spoke mode chips.**
   Highest-priority non-empty bucket (scorecard clean → bug found in today's Phase 1).
   The mobile top strip now wraps: switcher + stats pill share row 1, the mode chips own a
   full-width row 2. `width: 100%` on the chip row forces the wrap, so the chips can never sit under
   the pill at any width. Chips changed from fixed 30px squares to `flex: 1` with `min-height: 36px`,
   which also lifts them off the sub-40px touch-target floor, and gained `aria-pressed` so the
   selected mode is exposed to assistive tech. Both spokes, one commit.

Only one code change shipped deliberately: it is a P0 on the exact files another producer touched
hours ago, and stacking further polish on top would make a revert harder if needed.
