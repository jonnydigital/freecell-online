## Remaining backlog (surfaced this run)

Two **landscape-mobile** items for a future run that has true landscape tooling
(the CDP harness only emulates portrait — height derived as `width × 2.16` — and
the Chrome extension cannot reflow to a real mobile-landscape viewport, so
neither can currently render/verify landscape mode):

1. **Spoke landscape parity (U6).** FreeCell (`DomGameShell.tsx`) already handles
   landscape mobile (`innerWidth > innerHeight && innerHeight < 500`): it hides
   the top/bottom bars and renders a floating time/moves status pill + floating
   undo/redo. **Klondike and Spider shells have no landscape handling at all** —
   in landscape their persistent top+bottom bars eat the limited vertical space.
   Porting FreeCell's landscape treatment to the two spokes is the next U6 slice,
   but it needs landscape verification before shipping.
2. **FreeCell landscape status-pill a11y (U6, small).** The landscape floating
   status pill (`DomGameShell.tsx` ~L2104) is `aria-hidden="true"`, and there is
   no aria-live region for time/moves, so screen-reader users get **no** game
   number / time / moves in landscape (portrait exposes them via the top bar).
   Removing `aria-hidden` would restore parity — a one-line, low-risk change, but
   held today because landscape rendering can't be re-verified in this
   environment.

Longer-standing (unchanged): portrait dead-space is a planning signal, not a
failure — FreeCell ~46–48%, Klondike ~59–60% at phone widths (Klondike's larger
gap is inherent to its 7-pile layout). Candidate for below-board contextual
content per U6, tracked in the mobile-audit reports.
